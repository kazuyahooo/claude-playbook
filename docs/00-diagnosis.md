# 00. 快速診斷：本環境三大失效模式

version: 1.0 ｜ last-updated: 2026-07-09 ｜ 依據：抽讀 2026-07 的實際 session transcript（74 個 session，主體為兩個每日排程任務）

本檔是其他所有檔案的依據。修法的落點標在每條末尾。

## 失效模式 1：每個 session 從零重新推導狀態（最漏 token）

**證據**：兩個每日排程任務（Stock daily update、Daily learning digest）每次執行都重新做同樣的推導——學習任務要查 Notion 歷史才知道「昨天教到哪」、比對 195 個舊標籤去重；股票任務要查詢資料庫確認「本月快照是否已存在」「今天有無清倉配息事件」。這些推導每天重複付費，且隨歷史增長越來越貴。

**修法**：為每個排程任務建一個「狀態摘要」——Notion 頁面頂部一個固定區塊或資料庫欄位，格式固定、一眼可讀（例：`日文進度: じゃ行拗音(2026-07-09) | K8s: PromQL/histogram_quantile | 本月快照: 已建 07-08 | 已用標籤數: 205`）。每次執行開頭讀這個區塊代替查歷史，結尾更新它。查歷史只在狀態區塊缺失或自相矛盾時做。
**落點**：改 stock-daily-update 與 learning digest 的排程 prompt / skill（需使用者同意，屬 docs/04 的 L2 變更）。

## 失效模式 2：主對話自己下場做大量讀取與批次操作（最容易失焦）

**證據**：觀察到的 session 全部工作都在主對話 context 進行，數十次工具呼叫的原始回傳（Notion 頁面全文、bash 輸出、查價結果）全部堆在同一個 context。排程任務目前還撐得住，但互動 session 一旦涉及掃資料夾、讀多個網頁、批次改檔，主 context 會被原始資料塞爆，後半段開始忘記早期指令、重複已做過的事。

**修法**：大量讀取（>3 個檔案或網頁）、掃描、批次修改，一律派 subagent，主對話只收結論＋位置引用（檔案:行號 或 URL）。具體派法、模型選擇、回報合約見 docs/01；模板見 docs/03。
**落點**：docs/01-dispatch.md（已寫入）。

## 失效模式 3：踩過的坑不落地，下次重踩（最容易出錯）

**證據**：股票任務發現「VOO 交易均價與市價差距異常」「AMD 查無可靠報價來源」，寫進了當天的報告——但報告不會被下次執行讀到，同樣的異常明天會被當成新發現再查一次。同理：哪些 MCP connector 未授權、sandbox 的路徑對映、pip 要加 `--break-system-packages`，每個新 session 都可能重新試錯一遍。

**修法**：建立唯一的教訓落點 docs/05-environment.md（環境事實＋已知地雷清單），格式與寫入時機由 docs/04 規定。任何 session 遇到「花了超過兩次嘗試才弄對的事」就寫一條進去。
**落點**：docs/05-environment.md（已建，含目前已知全部地雷）＋ docs/04 的寫回協議。
