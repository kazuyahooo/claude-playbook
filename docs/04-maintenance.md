# 04. 維護協議＋交接信

version: 1.0 ｜ last-updated: 2026-07-09

## 1. 變更權限分級

- **L1 自行改，不必問**：往 docs/05 追加地雷條目；修正確定錯誤的事實（工具改名、路徑變了）；補充範例；更新各檔 version/last-updated。
- **L2 先問使用者再改**：刪除或反轉任何規則；改 CLAUDE.md 的絕對規則區；改排程任務的 prompt/skill；改驗收標準（docs/01 第 5 節、docs/02 第 5 節）。問法：說明想改什麼、為什麼、改了會有什麼行為差異。
- **L3 不可改**：本分級表自身、README 的誠實條款。要動它們，明確取得使用者逐字同意。

判斷原則：**放寬約束是 L2，收緊或補充是 L1**。拿不準就當 L2。

## 2. 教訓寫回（唯一落點：docs/05 的地雷清單）

觸發：花了超過兩次嘗試才弄對的任何事。當場寫，不要等 session 結尾（session 隨時可能中斷）。

格式（一條一行起，保持可掃讀）：

```
- [2026-07-09] {症狀}：{錯誤假設} → {正確做法}。（來源：{session 主題}）
```

範例：`- [2026-07-09] pip install 失敗：以為是網路問題 → sandbox 要加 --break-system-packages。（來源：股票更新）`

## 3. 精簡觸發條件

- CLAUDE.md > 120 行、或任一 docs 檔 > 200 行、或 docs/05 地雷 > 40 條 → 本 session 結束前執行精簡：合併同類條目、刪除已失效條目（環境已變的）、把長背景抽成新檔留一行指向。
- 精簡是 L1（收緊、去重），但**刪除仍然成立的規則是 L2**。
- 每季（或使用者要求時）做一次全面 re-probe：實測 docs/05 記載的工具與限制是否仍準確，過期就更新。

## 4. 給未來 session 的信

接手的模型你好。三件沒人問但你最需要知道的事：

**一、持久化只有一條路。** 你寫的任何檔案，除非進了這個 repo 且使用者 push，下個 session 就看不到。outputs 資料夾是幻覺般的持久化——它留在磁碟上但沒人會再讀。所以：值得留的東西寫進 playbook 對應檔案，session 結尾提醒 push。反過來說，你現在能讀到這些字，代表這條鏈路有效，請維護它。

**二、這個環境的重心是兩個每日排程任務。** 股票更新與學習摘要每天在跑、每天在花 token，對它們的任何改進（尤其 docs/00 失效模式 1 的狀態摘要方案，截至 2026-07-09 尚未實施）價值都會乘以 365。但改排程任務是 L2，先問使用者。

**三、使用者 wei 的工作風格。** 偏好極簡回覆；願意授權自主作業（開場問完就不想被打斷）；投資操作類的建議要保守——提供資訊與選項，不下指令（見股票任務裡「建議補現金、不需賣股」的口吻）。

**這套制度最可能的退化方式與預防：**

1. **膨脹**：每個 session 加一點，兩個月後 CLAUDE.md 變成沒人讀完的長文。預防：第 3 節的行數上限是硬的，超了就精簡，這不是建議。
2. **Cargo cult**：因為「這是 Fable 立的」而不敢改已經失效的規則。預防：每條規則都附了理由，理由不再成立就走 L2 流程改掉。建立者的身分不是論據。
3. **環境漂移**：工具改名、connector 授權變動、agent type 增減，讓 docs/01/05 的具體指令悄悄失真，弱模型會照著執行不存在的工具然後卡住。預防：發現不符當場記進 docs/05（L1），每季 re-probe。
4. **驗收儀式化**：read-back 淪為走過場，審查 agent 回「整體不錯」也被接受。預防：docs/03 T5 明文禁止空泛評語，驗收回報必須逐條有內容，否則重派。
