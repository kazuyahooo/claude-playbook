# 02. 判斷 rubric

version: 1.0 ｜ last-updated: 2026-07-09 ｜ 每條附正例（✅ 該觸發）與反例（❌ 不該觸發）。拿不準時，按最接近的例子走。

## 1. 何時升級模型（sonnet → opus）

判準：**失敗的原因是「想不對」而不是「做不完」**。推理錯誤、方案自相矛盾、抓不到問題本質 → 升級。只是量大、步驟多、要重跑 → 不升級，拆小或重派。

- ✅ 正例：除錯兩輪，每輪的修法都讓另一個測試壞掉——這是因果理解錯了，升 opus 重新診斷。
- ❌ 反例：改 40 個檔案改到一半 context 不夠——這是量的問題，拆成批次繼續用 sonnet。

## 2. 何時算真的完成

判準：**驗收條件逐條有證據**（測試輸出、read-back 回報、實跑結果），且證據來自非作者。三個都缺一不可：做了、驗了、驗的人不是做的人。

- ✅ 正例：改完排程 skill，手動觸發跑一次，輸出符合預期，且 fresh agent read-back 確認 prompt 無矛盾 → 完成。
- ❌ 反例：「檔案已寫入、內容我檢查過沒問題」——作者自查不算驗收，不算完成。

## 3. 何時停下來問使用者

判準：滿足任一即問：**（a）不可逆且影響範圍出 session**（刪資料、對外發送、花錢、改排程任務的行為）；**（b）兩個合理方案的選擇取決於使用者偏好而非事實**；**（c）升級模型後仍失敗**。問的時候附：已試什麼、卡在哪、建議選項與推薦。

- ✅ 正例：發現 Notion 股票資料庫有一筆疑似重複交易——刪除不可逆且可能是使用者故意的 → 問。
- ❌ 反例：不確定報告該用什麼標題——這有慣例可循，自己定一個講清楚即可，不要拿小事打斷使用者。

## 4. 什麼訊號代表方向錯了（換路，不是重試）

判準：**重試的修正量沒有在收斂**。第二次嘗試遇到的問題和第一次同類、或修 A 壞 B 修 B 壞 A、或為了讓方案成立而不斷加特例——方向錯了。換路 = 回到上一個分岔點選另一條，並把死路寫下來防止下個 agent 再走。

- ✅ 正例：用 regex 解析 Notion 匯出格式，每加一種頁面就要加新 pattern，第三次了——放棄 regex，改用結構化 API 拿資料。
- ❌ 反例：API 呼叫逾時一次——這是暫時性錯誤，重試即可，不是方向問題。

## 5. 品質底線怎麼驗

判準：每類產出有最低驗法，不可跳過：**文件** = fresh agent read-back（找矛盾與誤讀點）；**程式碼** = 至少實跑一次主路徑；**資料處理** = 抽 3 筆人工核對輸入輸出；**對外內容（email、發布）** = 使用者過目後才發。歸類原則：以敘述文字為主 → 文件；以逐筆數據為主 → 資料處理；混合體（如含數據的報告）兩種都做，取最小集（read-back 一次＋抽 3 筆）。產出是否「對外」要在動手前就判定，它決定最後多一道使用者過目關卡。

- ✅ 正例：算完持倉損益，抽台積電一檔手算核對——數字對得上才寫進報告。
- ❌ 反例：「腳本邏輯看起來對」就把 74 筆交易的計算結果直接寫進 Notion——沒抽驗，不合格。

## 6. 模糊題與品味判斷（誠實條款）

拆解和驗證補不了「這個方案雅不雅」「這篇文氣味對不對」這類判斷。遇到時按序處理：（1）把模糊處轉成可比較的選項，升 opus 或生成多版本評審選優；（2）仍無法收斂 → 給使用者 2–3 個具體選項附各自取捨，要第二意見；（3）都不行 → 明說「這題超出我能可靠判斷的範圍」，交代已做到哪。**禁止**：假裝有把握、或用「都可以」逃避。

- ✅ 正例：使用者要「更有質感」的簡報——生成三種風格截圖給使用者選，不自己猜。
- ❌ 反例：自行斷定某種配色「最有質感」然後做完整份——把品味題當事實題答。
