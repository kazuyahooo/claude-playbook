# 03. 派工 prompt 模板

version: 1.0 ｜ last-updated: 2026-07-09 ｜ 用法：複製對應模板，填 `{...}` 空格，經 Agent tool 發出。三件套（目標動機／驗收條件／回報格式）已內建，刪掉任何一段等於違反 docs/01 第 2 節。

## T1 搜尋／定位（agent: Explore，model: haiku 或省略）

```
目標：找出 {要找什麼，例：所有讀取 Notion 交易資料庫的程式碼}。
動機：{為什麼找，例：接下來要改資料庫 schema，需要知道影響面}。
範圍：{目錄/repo/網站}；搜尋廣度：{medium|very thorough}。
驗收：每個結果附 檔案:行號（或 URL）＋一句話說明為何相關；確定找完了要說明你搜了哪些關鍵字與路徑。
回報格式：條列，每條 = 位置 + 一句話。不要貼原始碼片段超過 3 行。找不到就回報「未找到」＋已搜範圍，不要猜。
```

## T2 實作（agent: general-purpose，model: sonnet）

```
目標：{做什麼，例：為 X 腳本加上 Y 功能}。
動機：{為什麼，例：使用者每天手動做這步，要自動化}。
現況：{相關檔案路徑、既有行為、已知限制}。
約束：{不可動的東西；需遵守的慣例；改既有檔案前先備份 .bak-<日期>}。
驗收（逐條可檢查）：
- {例：執行 `python x.py --dry-run` 輸出包含 Z}
- {例：原有功能 W 行為不變，實跑證明}
回報格式：改了哪些檔案（路徑清單）、每個驗收條件的實跑證據（指令＋輸出摘要）、未解決事項。程式碼不要貼回，落檔傳路徑。
```

## T3 重構（agent: general-purpose，model: sonnet；跨模組大重構用 opus）

```
目標：{重構什麼、成什麼樣}。
動機：{現在的痛點}。
不變式（最重要）：重構前後外部行為必須完全一致，以 {測試套件/驗證腳本} 為準。沒有現成測試就先寫最小行為快照再動手。
範圍：{哪些檔案可動、哪些絕對不可動}。
驗收：{測試/腳本} 重構前後皆通過；diff 中不含行為變更（只有結構變更）。
回報格式：變更摘要（動了什麼、為什麼）、驗證證據、任何你發現但沒動的問題（列出來即可）。
```

## T4 研究（agent: general-purpose，model: sonnet）

```
目標：回答 {具體問題，例：2026 年 X 服務的 API 限額與定價}。
動機：{這答案要用來做什麼決定}。
方法要求：用 WebSearch / web fetch 工具查當下資料（工具若在 deferred 清單，先用 ToolSearch 載入 schema 再呼叫；實際工具名以環境為準，如 mcp__workspace__web_fetch）；不要用訓練知識回答時效性問題；每個關鍵事實附來源 URL；來源之間矛盾要指出，不要擅自調和。
驗收：問題的每個子項都有附來源的答案，或明確標註「查不到」。
回報格式：結論先行（≤10 行），然後逐項事實＋來源。長篇整理落檔傳路徑，不要全文貼回。
```

## T5 審查／驗收（agent: general-purpose，model: sonnet；審 opus 產物用 opus）

```
你是 fresh-context 審查者，與產出者無關，你的價值在於沒有他的假設。
目標：審查 {檔案路徑清單 / diff}。
審查判準（逐條回報，沒問題也要說「此項通過」）：
- 內容完整：有沒有寫到一半、引用不存在的檔案或工具
- 內部矛盾：規則之間、文件與程式之間互相打架的地方
- 誤讀風險：一個沒有上下文的讀者會誤解的句子（引用原句＋你的誤讀版本）
- {任務特定判準，例：安全性、效能}
回報格式：問題清單，每條 = 位置 + 問題 + 建議修法，按嚴重度排序。禁止空泛評語（「整體不錯」不算審查結果）。
```
