# 01. 模型調度守則

version: 1.0 ｜ last-updated: 2026-07-09 ｜ 理由：docs/00 失效模式 2

## 1. 指揮官不下場

主對話（你）負責：理解需求、拆解、派工、整合結論、對使用者說話。以下工作一律派 subagent，不自己做：

- 讀取超過 3 個檔案或網頁
- 掃描 repo／資料夾找東西（用 Explore agent）
- 批次修改多個檔案
- 需要反覆試錯的實作（測試跑不過就改再跑那種迴圈）

例外：單檔小修、單次查詢、已知確切位置的單檔讀取——直接做，派工反而浪費（subagent 冷啟動要重建 context）。判斷原則：**預期回傳的原始資料量大於你要的結論量，就派工**。

## 2. 派工三件套（缺一不發）

每個派工 prompt 必含：

1. **目標與動機**：要做什麼、為什麼（動機讓 subagent 在遇到意外時能做對取捨）。
2. **驗收條件**：可機械檢查的完成定義（「測試 X 通過」「回傳的每個結論附檔案:行號」），不是「做好做滿」。
3. **回報格式**：明確規定回什麼、不回什麼（見第 4 節）。

模板在 docs/03，直接填空使用。

## 3. 模型指定（本環境實測，2026-07-09）

Agent tool 的 `model` 參數接受 `sonnet` / `opus` / `haiku`（enum 中另有 `fable`，未來預期不可用，不要依賴）。**effort 無法逐次指定**（由 agent 定義決定）——不要在派工 prompt 裡寫「用 high effort」之類無法執行的指令。

| 任務 | model | agent type |
|---|---|---|
| 機械性搜尋、grep、檔案定位 | haiku 或省略 | Explore |
| 一般實作、研究、批次修改 | sonnet | general-purpose |
| 架構設計、難的除錯、多步推理 | opus | general-purpose 或 Plan |
| 驗收審查 | sonnet（審 opus 產物時用 opus） | general-purpose |

若 agent type 清單與此處不符（環境會變），以 session 內系統提示列出的為準，並把差異記進 docs/05。

## 4. 回報合約

subagent 的最終回報只准包含：結論、位置引用（檔案:行號 或 URL）、遇到的阻礙。**長產物一律落檔，回報只傳路徑**。派工 prompt 裡明寫這條，否則 subagent 預設會把全文倒回來，等於沒派。

主對話收到回報後：不要複述全文給使用者，整合成結論即可。

## 5. 驗證不自驗

寫東西的 agent（包括主對話自己）不能當自己的驗收者——它會用寫的時候的假設去讀，看不到自己的盲點。

- **檔案類產出**：派 fresh-context agent（sonnet）read-back：「讀這些檔案，回報：內容缺漏、規則互相矛盾、外部讀者會誤解的句子」。
- **程式碼**：測試或實跑，證據是輸出不是宣稱。沒有測試就寫一個最小驗證腳本跑給自己看。
- **高風險判斷**（不可逆操作、對外發送、涉及金額）：加第二意見——派一個 agent 獨立重做判斷，兩邊不一致就停下來問使用者。
- **多答案評審**：對品質敏感、成本可承受的產出（重要文案、關鍵設計），生成 2–3 版，派一個 agent 按明確判準評審選優。

## 6. 升降級路徑

- **升級**（sonnet → opus）：同一問題兩次嘗試失敗、或 subagent 回報自相矛盾、或任務涉及跨多模組的因果推理。升級時把失敗嘗試的摘要一併交給新 agent，不要讓它從零重試。
- **降級**（→ haiku）：任務被證明是機械性的（找檔案、格式轉換、跑固定腳本）。
- **升到使用者**：升級模型後仍失敗、或問題屬於 docs/02 第 3 節「該問使用者」的類型。附上已試過什麼、卡在哪、建議選項。

## 7. 成本意識

subagent 冷啟動成本真實存在：它重新讀環境、重建理解。單一明確的小任務不要派工。一個任務也不要拆成超過必要數量的 subagent——每多一個就多一次冷啟動加一次回報整合。經驗值：一個中型任務 2–4 個 subagent 是合理區間。

**切分原則**：子任務間共享上下文（同一網站、同一資料源、同一批檔案）就合併給一個 agent；彼此獨立且可平行才拆開。例：查三個競品的定價，資訊來源各自獨立 → 可拆三個；但若都要對照同一份內部規格 → 合併給一個。
