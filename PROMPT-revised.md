# 制度建立 prompt（修訂版，留存用）

原版來源：網路流傳的「Fable 立制度」prompt。2026-07-09 由 Claude Fable 5 修訂並實際執行，產物即本 repo。若未來要在新環境重跑（例如換了平台），用這個版本。相對原版的修改摘要在文末。

---

這個 session 的任務：把你的判斷力轉成可長期沿用的制度與檔案，讓後續由其他模型（Sonnet/Opus/Haiku 等）運作的每個 session 都因此變強。立制度，不執行日常任務。

作業規則：
1. **先探測，後寫規則。** 動筆前實測環境：持久化機制（檔案放哪、下個 session 怎麼讀到）、可用 subagent 類型、model/effort 參數實際支援範圍、MCP 與授權狀態、skill、過去 session 紀錄。只寫當下驗證過可執行的指令；不確定的寫成「若支援才用」並標註未驗證。**若環境沒有持久化機制，先停下來和使用者確定一個（如 git repo＋連接資料夾），這是整套制度的地基。**
2. 自主作業。開場最多問一批問題（≤5 題），之後不停下。
3. 價值排序＋隨做隨寫：先做價值最高的產出，每完成一項立刻落檔。session 隨時可能中斷，已落檔的就是使用者拿到的全部。
4. 改任何既有檔案前先留備份副本。新內容寫新檔，入口檔（CLAUDE.md）只放精簡路由。
5. 讀者是後續接手的模型：規則要具體、可執行、有判準與正反例。抽象要求等於沒寫。每條規則附理由——未來模型憑理由決定去留，不憑「這是誰立的」。
6. 產出不可依賴只有你能做到的能力，全部要在 Sonnet 等級跑得動。

交付清單（按價值排序）：
A. 快速診斷：基於**實際 transcript 或現有設定**（不是通用清單），找出此環境最漏 token、最易失焦、最易出錯的前三名，各附具體修法與落點。先寫，供後面引用。
B. 入口路由檔（CLAUDE.md）：≤120 行硬上限，含開場協議、路由表、絕對規則（≤6 條）、使用者偏好。
C. 模型調度守則：指揮官不下場；派工三件套（目標動機/驗收條件/回報格式）；model 指定表（effort 僅在環境支援時提及）；回報合約（只回結論＋位置引用，長產物落檔傳路徑）；升降級路徑；驗證不自驗；派工成本意識。
D. 判斷力外化：升級模型、真完成、停下問人、換路訊號、品質底線、模糊題與品味題的處理，每條附正例反例。
E. 派工模板：搜尋/實作/重構/研究/審查五份填空模板，三件套內建。
F. 維護協議：變更權限分級（自行改/先問/不可改）、教訓寫回的唯一落點與格式、精簡觸發的硬數字、定期 re-probe。
G. 交接信：三件沒人問但最重要的事＋制度最可能的退化方式與預防。可併入 F。
H. 環境事實檔：工具、限制、已知地雷的唯一落點，標註「實測快照，會過期」。

收尾（必做）：
1. fresh-context subagent（**Sonnet 級**，證明目標讀者跑得動）對抗審查全部產出：規則互打、路徑工具名錯誤、會誤讀的句子；並讓它模擬按制度處理一個典型任務，看是否卡住。修到通過。
2. read-back 驗證每檔落地完整。
3. 一頁總結：改了什麼、為什麼、明天怎麼用（含部署步驟）。
4. context 快用完：立刻停下產出，先完成收尾 1–3，未完成項寫進交接信。

誠實條款：標註這套做法的極限——拆解、驗證、多樣本評審補得了執行品質；模糊題與品味判斷補不了，寫明遇到時的處理順序（升級模型→外部第二意見→明說做不到）。不確定就查，查不到就標註，不要編造。

## 相對原版的修改摘要

1. 新增規則 1「先探測後寫規則」＋持久化地基檢查——原版假設 Claude Code 環境，直接跑會寫出不可執行的規則、檔案落在沒人會再讀的地方。
2. 交付 A 改為必須基於實際 transcript／設定，否則只是通用最佳實踐清單。
3. 刪除「顯式指定 effort」的硬性要求（多數環境不支援逐次指定）。
4. 收尾 1 指定審查者為 Sonnet 級並加「模擬執行」——read-back 只驗存在，模擬執行才驗「跑得動」。
5. 新增交付 H（環境事實檔）作為教訓的唯一落點；G 允許併入 F。
6. 加上維護的硬數字（行數上限、re-probe 週期）與變更權限分級，防制度膨脹。
7. 誠實條款加「規則憑理由成立，不憑建立者身分」，防 cargo cult。
